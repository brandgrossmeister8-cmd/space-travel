## КАК ПОЛЬЗОВАТЬСЯ ЭТИМ ШАБЛОНОМ

> *Этот блок — только для понимания. В реальных ТЗ его удалить.*
>
> **Что такое ТЗ?** Техническое задание — документ-контракт между заказчиком и исполнителем. По нему подрядчик делает работу, по нему принимаете результат. Чем точнее ТЗ — тем меньше споров и переделок.
>
> **Правило:** Если после прочтения ТЗ у подрядчика возникают вопросы — ТЗ неполное.
>
> **Пояснения в шаблоне** выделены курсивом в блоках `>`. При создании реального ТЗ все пояснения удаляются, остаётся только содержание.

## 4. ЭКОНОМИКА (ROI)

> *Зачем: Обоснование инвестиций. Руководство видит, когда вернутся деньги.*
>
> *ROI (Return on Investment) = возврат инвестиций. Показывает, сколько заработаем на каждый вложенный рубль.*
>
> *Формула: ROI = (Экономия − Затраты) ÷ Затраты × 100%*

| Показатель | Значение | Как считали |
|------------|----------|-------------|
| Экономия времени | XX ч/мес | [Формула] |
| Экономия в деньгах | XXX 000 ₽/год | XX ч × XXX ₽/ч × 12 мес |
| Затраты (разово) | XXX 000 ₽ | Разработка + внедрение |
| Затраты (ежемесячно) | XX 000 ₽/мес | Подписки, поддержка |
| **Окупаемость** | **~X мес** | Затраты ÷ Экономия/мес |
| **ROI (год 1)** | **+XX%** | (Экономия − Затраты) ÷ Затраты |

## 8. ИНТЕГРАЦИИ

>
> *Интеграция = автоматический обмен данными между системами.*

### 8.1. Какие системы участвуют

| Система | Версия | Роль | Способ связи |
|---------|--------|------|--------------|
| [Система 1] | X.X | Источник данных | REST API |
| [Система 2] | X.X | Получатель | Webhook |
| [Инструмент интеграции] | — | Связующее звено | — |

> *Термины:*
> *— API (Application Programming Interface) = способ «общения» между программами*
> *— REST API = самый распространённый тип API, работает через интернет*
> *— Webhook = система сама сообщает о событии (не нужно постоянно спрашивать)*

### 8.2. Потоки данных

> *Зачем: Наглядная схема — откуда куда что течёт.*

```
│  Система A  │────────►│  Интеграция │────────►│  Система B  │
│  (источник) │         │  (Albato)   │         │ (получатель)│
│             │         │             │         │             │
└─────────────┘         └─────────────┘         └─────────────┘
```

**Таблица потоков:**

| Откуда | Куда | Какие данные | Когда | Как часто |
|--------|------|--------------|-------|-----------|
| [Система A] | [Система B] | [Что передаём] | [Событие или расписание] | [Real-time / X раз в день] |

> *Real-time (реальное время) = мгновенно, как только произошло событие*

> *Зачем: Конкретные «адреса» и форматы для программиста. Без этого он будет угадывать.*
>
> *Если API ещё не изучен — написать «Будет получено после выполнения P-01 (аудит API)»*

#### Пример запроса: Получить данные

| Параметр | Значение |
|----------|----------|
| Метод | GET / POST / PUT / DELETE |
| URL | `https://api.система.ru/v1/ресурс/{id}` |

**Пример ответа (JSON):**
```json
{
  "id": 12345,
  "name": "Пример названия",
  "status": "active",
  "created_at": "2026-02-19T14:30:00Z"

```

> *JSON (JavaScript Object Notation) = формат данных, похож на структурированный текст. Фигурные скобки = объект, квадратные = список.*

**Коды ответов (что означают):**

| Код | Что значит | Что делать |
|-----|------------|------------|

### 8.5. Обработка ошибок

> *Зачем: Что делать, если что-то пошло не так. Без этого система упадёт и никто не узнает.*
>
> *Retry = повторная попытка*
> *Fallback = запасной вариант*

| Тип ошибки | Повторять? | Через сколько | Сколько раз | Если не помогло |
|------------|------------|---------------|-------------|-----------------|
| Нет ответа (timeout) | Да | 5 сек → 30 сек → 2 мин | 3 | Очередь + алерт IT |
| Ошибка сервера (5xx) | Да | 5 мин | 5 | Очередь + алерт IT |
| Неверные данные (4xx) | Нет | — | 1 | Лог + алерт владельцу |
| Нет интернета | Да | 1 мин | 10 | Очередь + алерт IT |

**Куда отправлять алерты:**

| Событие | Куда | Кому |
|---------|------|------|
| Много ошибок (>5/час) | Telegram | IT-чат |
| Критическая ошибка | Email + звонок | Логвинов Г.А. |
| Очередь переполнена | Email | Владелец процесса |

> *— C (Consulted) = консультирует*

| Этап | Делает (R) | Отвечает (A) | Консультирует (C) |

> *UAT (User Acceptance Testing) = приёмочное тестирование пользователями. Сотрудники проверяют, что система делает то, что нужно.*

| 3 | Данные корректны | 100% | Сверка выборки |
| 4 | Производительность в норме | По п.9.1 | Замеры |
| 5 | Документация готова | Да | Наличие файлов |

### 11.3. Процедура приёмки

1. **Демонстрация** — подрядчик показывает работу системы (1-2 часа)
2. **Тестирование** — заказчик проверяет тест-кейсы (1-3 дня)
3. **Протокол** — список замечаний (если есть)
4. **Исправление** — подрядчик устраняет замечания
5. **Повторная проверка** — проверяем исправления
6. **Акт приёмки** — подписание при выполнении критериев

## 12. ПОДДЕРЖКА И ГАРАНТИИ

> *Зачем: Что делать после внедрения, если что-то сломается.*

### 12.1. Гарантийный период

- **Срок:** X месяц(а) после подписания акта
- **Что входит:** исправление ошибок, консультации
- **Что НЕ входит:** новые функции, изменение требований

### 12.2. SLA (уровень сервиса)

> *SLA (Service Level Agreement) = договор об уровне обслуживания. Фиксирует, как быстро чинят проблемы.*

| Приоритет | Что случилось | Время реакции | Время решения |
|-----------|---------------|---------------|---------------|
| P1 Критический | Система не работает | 1 час | 4 часа |
| P2 Высокий | Важная функция не работает | 4 часа | 1 рабочий день |
| P3 Средний | Функция работает с ошибками | 1 рабочий день | 3 рабочих дня |
| P4 Низкий | Мелкие проблемы, пожелания | 3 рабочих дня | По согласованию |
