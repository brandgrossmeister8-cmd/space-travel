# Экспертные рекомендации по разработке ТЗ на автоматизацию

**Дата:** 2026-02-18
**Задача:** 4.4 Разработка ТЗ — подготовительный этап
**Для кого:** PM, IT-отдел, подрядчики

---

## Резюме для руководства

Перед созданием ТЗ проведён анализ лучших практик от экспертов по автоматизации. Ключевые выводы:

1. **ТЗ ≠ Требования.** Требования описывают «что нужно бизнесу», ТЗ — «как это реализовать технически»
2. **Главная причина провалов** — нечёткие формулировки и отсутствие критериев приёмки
3. **Для туризма** критичны: двухвалютность, интеграция с GDS/САМО, шаблоны договоров, тайм-лимиты
4. **Стандарт** — ГОСТ 34.602-2020 (обновлённая версия ГОСТ 34.602-89)

## 1. Критерии эффективного ТЗ

### 1.1. Что делает ТЗ эффективным (чек-лист качества)

На основе [лучших практик от Сбера](https://developers.sber.ru/help/business-development/technical-specification), [X5 Tech](https://habr.com/ru/companies/X5Tech/articles/744888/) и [международных стандартов](https://www.pandium.com/blogs/what-to-include-in-an-integration-requirements-document-template-included):

| # | Критерий | Проверочный вопрос | Плохо | Хорошо |
|---|----------|-------------------|-------|--------|
| 1 | **Измеримость** | Можно ли проверить, что требование выполнено? | «Система должна быстро работать» | «Время отклика API < 2 сек при 100 запросах/мин» |
| 2 | **Однозначность** | Может ли разработчик понять по-разному? | «Удобный интерфейс» | «Форма содержит 3 поля: ФИО, телефон, email» |
| 3 | **Полнота** | Описаны ли ВСЕ сценарии? | Только основной сценарий | Основной + альтернативные + ошибки |
| 4 | **Непротиворечивость** | Нет ли конфликтов между требованиями? | «Данные хранятся вечно» + «GDPR: удалять через 3 года» | Явно указаны правила удаления |
| 5 | **Трассируемость** | Можно ли понять, откуда требование? | Требование без обоснования | «FR-01 реализует AUTO-035 для решения проблемы X» |
| 6 | **Приоритизация** | Ясно ли, что обязательно, а что опционально? | Все требования одинаковые | Must Have / Should Have / Could Have |
| 7 | **Тестируемость** | Есть ли тест-кейсы? | Только описание функции | Тест-кейс: «При вводе X → ожидаем Y» |
| 8 | **Визуализация** | Есть ли схемы и диаграммы? | Только текст | Диаграммы потоков данных, макеты UI |

### 1.2. Формула SMART для требований

Каждое требование должно быть:

- **S**pecific (конкретное): чёткое описание функции
- **M**easurable (измеримое): количественные показатели
- **A**chievable (достижимое): реализуемо в рамках бюджета/сроков
- **R**elevant (релевантное): решает бизнес-задачу
- **T**ime-bound (ограниченное по времени): есть дедлайн

## 2. Типичные ошибки в ТЗ (и как их избежать)

На основе [анализа от B-152](https://b-152.ru/osnovnye_oshibki_napisaniya_tz_na_sozdanie_informacionnoj_sistemy) и [Program Store](https://wiki.programstore.ru/kak-sostavit-tekhnicheskoe-zadanie-kotoroe-pojmut-vse/):

### 2.1. ТОП-10 ошибок

| # | Ошибка | Последствия | Как исправить |
|---|--------|-------------|---------------|
| 1 | **Нет структуры** — хаотичный список требований | Разработчик теряет время на поиск информации | Использовать единый шаблон по ГОСТ 34.602 |
| 2 | **Нет целевой аудитории** — только пожелания заказчика | Продукт не решает задачи пользователей | Описать персоны и сценарии использования |
| 3 | **Нет критериев приёмки** — «сделайте хорошо» | Споры при сдаче проекта | Прописать тест-кейсы для каждого требования |
| 4 | **Нет альтернативных сценариев** — только happy path | Система падает при ошибках | Описать: что если данные неполные? API недоступен? |
| 5 | **Забыли админку** — только front-end | Нет инструментов управления | Чек-лист: роли, настройки, логи, отчёты |
| 6 | **Нет требований к нагрузке** — «должно работать» | Система падает при росте | Указать: сколько пользователей, записей, запросов/сек |
| 7 | **Нет требований к интеграции** — «подключить к 1С» | Интеграция не работает | Детально: API, формат данных, частота, обработка ошибок |
| 8 | **Размытые сроки** — «как можно скорее» | Проект затягивается | Фиксированные этапы с датами |
| 9 | **Нет бюджета** — «оценим потом» | Переделки, конфликты | Бюджет = фиксатор scope |
| 10 | **Нет ответственных** — «кто-нибудь сделает» | Никто не делает | RACI-матрица для каждого этапа |

### 2.2. Антипаттерны формулировок

| Плохо | Почему плохо | Хорошо |
|-------|--------------|--------|
| «Интуитивно понятный интерфейс» | Субъективно, нельзя проверить | «Новый пользователь создаёт заявку за < 3 мин без обучения» |
| «Высокая производительность» | Нет цифр | «Обработка 1000 записей за < 10 сек» |
| «Интеграция с САМО» | Не ясно, какие данные, как часто | «Синхронизация карточек агентов (ID, название, лимит) каждые 5 мин через САМО API» |
| «Уведомления о важных событиях» | Какие события? Куда уведомления? | «Email-уведомление менеджеру при превышении лимита агента на > 10%» |
| «Безопасная передача данных» | Не ясно, какой стандарт | «TLS 1.3, OAuth 2.0, логирование всех запросов» |

## 3. Структура ТЗ по ГОСТ 34.602-2020

На основе [официального ГОСТ 34.602](https://docs.cntd.ru/document/1200006924) и [практического руководства](https://habr.com/ru/articles/769648/):

### 3.1. Обязательные разделы

```
1. Общие сведения
   1.1. Наименование системы
   1.2. Заказчик и исполнитель
   1.3. Основания для разработки
   1.4. Сроки начала и окончания работ

2. Назначение и цели создания системы
   2.1. Назначение системы
   2.2. Цели создания (измеримые!)

3. Характеристика объекта автоматизации
   3.1. Краткие сведения об объекте
   3.2. Условия эксплуатации

4. Требования к системе
   4.1. Требования к структуре и функционированию
   4.2. Требования к функциям (задачам)
   4.3. Требования к видам обеспечения
       - Информационное обеспечение
       - Программное обеспечение
       - Техническое обеспечение
       - Организационное обеспечение

5. Состав и содержание работ
   5.1. Перечень этапов
   5.2. Сроки и ответственные

6. Порядок контроля и приёмки
   6.1. Виды испытаний
   6.2. Критерии приёмки

7. Требования к документированию

8. Источники разработки (приложения)
```

### 3.2. Адаптация ГОСТ для IT-проектов

Для интеграционных проектов Space Travel рекомендуется добавить:

| Раздел ГОСТ | Дополнение для интеграций |
|-------------|---------------------------|
| 4.2 Функции | User Stories в формате «Как [роль], я хочу [действие], чтобы [результат]» |
| 4.3 Информационное | Диаграмма потоков данных, формат JSON/XML, примеры payload |
| 4.3 Программное | Требования к API: endpoints, методы, авторизация, rate limits |
| 5. Работы | Agile-спринты вместо водопада (для гибкости) |
| 6. Приёмка | Автотесты + ручное тестирование + UAT (приёмка пользователями) |

## 4. Специфика ТЗ для туристического бизнеса

На основе [САМО-Софт](https://samo.ru/), [U-ON.Travel](https://u-on.ru/travel/) и [отраслевых обзоров](https://upr.ru/article/avtomatizaciya-turisticheskogo-biznesa/):

### 4.1. Что обязательно учесть

| Аспект | Почему критично | Как отразить в ТЗ |
|--------|-----------------|-------------------|
| **Двухвалютность** | Туры в USD/EUR, расчёты в RUB | Явно указать: курс ЦБ или договорной, дата фиксации |
| **Тайм-лимиты авиа** | Билеты «сгорают» — прямые убытки | Алерты за 48/24/6 часов, эскалация на руководителя |
| **Договоры из шаблонов** | Юридические риски | Автозаполнение реквизитов поставщика, проверка актуальности |
| **Интеграция с GDS** | Amadeus, Galileo, Sabre — разные API | Явно указать: какой GDS, NDC или legacy, формат PNR |
| **ЕФРТА / Турпомощь** | Регуляторные требования | Проверка статуса агента, регистрация туристов |
| **Сезонность** | Пиковые нагрузки в сезон | Требования к масштабируемости: +300% к базовой нагрузке |

### 4.2. Интеграции туристических систем

| Система | API | Особенности | Что указать в ТЗ |
|---------|-----|-------------|------------------|
| САМО-тур | REST API (ограниченный) | Не все методы документированы | Провести P-01 (аудит API) до начала разработки |
| Битрикс24 | REST API | Хорошо документирован | Указать версию API, права доступа, webhook-и |
| 1С | OData / COM / Файлы | Зависит от конфигурации | Явно: версия 1С, способ обмена, формат файлов |
| GDS | NDC / EDIFACT | Сертификация, песочница | Указать: тестовая среда, credentials, формат ответов |
| Юнисендер | REST API | Лимиты рассылок | Rate limit, формат контактов, статусы доставки |

## 5. Идеальный конечный результат (ИКР) технического задания

### 5.1. Определение ИКР

**Идеальное ТЗ** — документ, по которому разработчик может:
1. Начать работу без дополнительных вопросов
2. Оценить трудоёмкость с точностью ±20%
3. Написать автотесты ДО начала разработки
4. Сдать проект с первого раза без споров

### 5.2. Чек-лист «Готово ли ТЗ к передаче разработчику?»

```
ОБЩИЕ ТРЕБОВАНИЯ
□ Есть измеримая цель проекта (KPI до/после)
□ Определён scope (что входит, что НЕ входит)
□ Указан бюджет и сроки
□ Назначены ответственные (RACI)

ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
□ Описаны ВСЕ user stories (кто, что, зачем)
□ Для каждой функции есть критерий приёмки
□ Описаны альтернативные сценарии и ошибки
□ Есть приоритеты (Must/Should/Could)

ИНТЕГРАЦИИ
□ Указаны ВСЕ системы-участники с версиями
□ Для каждой интеграции: источник → получатель → данные → частота
□ Описан формат данных (JSON/XML + примеры)
□ Описана обработка ошибок (retry, алерты, fallback)
□ Указаны ограничения API (rate limit, размер payload)

НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
□ Производительность (время отклика, пропускная способность)
□ Надёжность (uptime, резервирование)
□ Безопасность (авторизация, шифрование, логирование)
□ Масштабируемость (текущая + планируемая нагрузка)

ТЕСТИРОВАНИЕ И ПРИЁМКА
□ Описаны тест-кейсы для ключевых сценариев
□ Определены критерии успешной приёмки
□ Указан период опытной эксплуатации
□ Описан процесс передачи в эксплуатацию

ПРИЛОЖЕНИЯ
□ Диаграммы (архитектура, потоки данных)
□ Макеты UI (если есть интерфейс)
□ Примеры API-запросов и ответов
□ Глоссарий терминов
```

### 5.3. Метрики качества ТЗ

| Метрика | Целевое значение | Как измерить |
|---------|------------------|--------------|
| **Вопросов от разработчика** | < 5 на 100 требований | Считать вопросы в первые 3 дня |
| **Точность оценки** | ±20% от плана | Факт/План по трудоёмкости |
| **Изменений scope** | < 10% в ходе проекта | Change requests / Total requirements |
| **Успешность приёмки** | 90%+ с первого раза | Тесты пройдены / Тесты всего |
| **Удовлетворённость** | > 8/10 | Опрос заказчика и исполнителя |

## 6. Рекомендуемый шаблон ТЗ для Space Travel

На основе анализа лучших практик, предлагается следующая структура:

### 6.1. Структура шаблона

```
1. ПАСПОРТ ПРОЕКТА (1 страница)
   - Название, версия, дата
   - Заказчик, исполнитель
   - Бюджет, сроки
   - Связанные AUTO/DASH точки

2. БИЗНЕС-КОНТЕКСТ (1-2 страницы)
   - Проблема (AS-IS с цифрами)
   - Цель (TO-BE с KPI)
   - Ожидаемый эффект (ROI)

3. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ (основная часть)
   - User Stories в формате: «Как [роль], я хочу [действие], чтобы [результат]»
   - Для каждой: приоритет, критерий приёмки
   - Альтернативные сценарии

4. ТРЕБОВАНИЯ К ИНТЕГРАЦИИ (для интеграционных проектов)
   - Системы-участники
   - Потоки данных (таблица + диаграмма)
   - API-спецификации (формат, примеры)
   - Обработка ошибок

5. НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ
   - Производительность
   - Безопасность
   - Масштабируемость

6. ЭТАПЫ РЕАЛИЗАЦИИ
   - Спринты / Этапы с датами
   - Зависимости
   - Риски и митигация

7. ПРИЁМОЧНЫЕ ИСПЫТАНИЯ
   - Тест-кейсы (таблица)
   - Критерии успеха
   - Процедура приёмки

ПРИЛОЖЕНИЯ:
A. Диаграмма архитектуры
B. Примеры API (запрос/ответ)
C. Макеты UI (если есть)
D. Глоссарий
```

### 6.2. Особенности для Space Travel

| Раздел | Специфика Space Travel |
|--------|------------------------|
| Паспорт | Указать связь с AUTO-XXX / DASH-XX из реестра |
| Бизнес-контекст | Ссылка на 3.4_automation_potential.md (экономия ч/мес) |
| Интеграции | Обязательно: САМО API status (P-01), 1С connector (P-05) |
| Нефункциональные | Учесть сезонность: пик в мае-августе (+300% нагрузка) |
| Приёмка | Пилот на 1 департаменте → масштабирование |

## 7. Процесс разработки ТЗ

### 7.1. Этапы

```
Этап 1: Сбор требований (1-2 дня)
├── Интервью с владельцем процесса
├── Анализ документации (AUTO/DASH описания)
└── Определение scope и границ

Этап 2: Проектирование (2-3 дня)
├── Диаграмма архитектуры
├── Описание потоков данных
└── Черновик функциональных требований

Этап 3: Детализация (2-3 дня)
├── User Stories
├── Требования к API
└── Тест-кейсы

Этап 4: Согласование (1-2 дня)
├── Review с IT
├── Review с бизнесом
└── Финализация
```

### 7.2. RACI для разработки ТЗ

| Активность | PM | Бизнес-аналитик | IT-архитектор | Заказчик |
|------------|----|-----------------|--------------| ---------|
| Сбор требований | A | R | C | I |
| Проектирование архитектуры | I | C | R | I |
| Написание ТЗ | A | R | C | I |
| Согласование | R | I | I | A |

**R** = Responsible, **A** = Accountable, **C** = Consulted, **I** = Informed

## 8. Выводы и рекомендации

### 8.1. Ключевые принципы

1. **Измеримость важнее красоты** — лучше «< 2 сек», чем «быстро»
2. **Примеры важнее описаний** — JSON-payload лучше абстракций
3. **Тест-кейсы = контракт** — что можно протестировать, то и сдадим
4. **Scope = бюджет** — что не в ТЗ, то за отдельные деньги

### 8.2. Рекомендации для Space Travel

| # | Рекомендация | Обоснование |
|---|--------------|-------------|
| 1 | **Провести P-01 (аудит САМО API) до написания ТЗ** | Многие требования зависят от возможностей API |
| 2 | **Использовать единый шаблон** | Ускоряет согласование, снижает риск ошибок |
| 3 | **Включать примеры данных** | Разработчик видит реальные значения |
| 4 | **Согласовывать с IT до финализации** | IT оценит реализуемость |
| 5 | **Версионировать ТЗ** | История изменений = защита от споров |

### 8.3. Следующие шаги

1. ✅ Утвердить структуру шаблона ТЗ
2. ⬜ Создать шаблон `TEMPLATE_technical_specification.md`
3. ⬜ Разработать TZ_01: САМО ↔ Битрикс24
4. ⬜ Разработать TZ_02: Автоматизация рутины
5. ⬜ Разработать TZ_03: Единая система отчётности (BI)
